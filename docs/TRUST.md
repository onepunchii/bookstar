# 신뢰 & 사기 방지 설계 (Trust Layer)

전제: 연예인 섭외 시장은 사칭 사기가 실존하는 시장. 사기 1건 = 플랫폼 신뢰 전체 훼손.
역으로 이 레이어가 완성되면 "검증된 직거래"가 BOOKSTAR의 핵심 상품이 된다.

## 공격 시나리오

| # | 시나리오 | 심각도 |
|---|---|---|
| A1 | 유령 소속사가 유명 아티스트 도용 등록 → 계약금 수취 후 잠적 | 치명 |
| A2 | 실사업자이지만 전속 관계 없는 무권한 브로커 행세 | 높음 |
| A3 | 타인 사진·경력 도용한 가짜 아티스트 프로필 | 중간 |
| A4 | 가짜 광고주의 허위 행사·출연료 미지급 (역방향) | 높음 → 주최자 검증 배지로 대응 중 |

## 방어 레이어

### L1. 입점 검증 (KYB)
- 사업자등록번호 → 국세청 진위확인·휴폐업 상태조회 API (공공데이터포털, 무료) 자동 검증
- **대중문화예술기획업 등록번호 필수** (대중문화예술산업발전법상 매니지먼트업 등록 의무)
  → 한국콘텐츠진흥원 등록부 대조. 개인의 무자격 소속사 등록 원천 차단
- 법인 계좌 1원 인증 (계좌 실명 = 사업자명 일치)
- 대표자 휴대폰 본인인증

### L2. 아티스트-소속사 관계 증명 (핵심)
- **아티스트 본인 승인**: 등록 시 아티스트 휴대폰으로 승인 요청 → 승인 전 비공개
  (기존 3-역할 구조의 아티스트 모드가 승인 주체)
- 유명인 등록은 무조건 수동 검수 큐 (포털 인물정보·소속사 공식 채널 교차 대조)
- 신뢰 3단계:
  1. `unverified` — 검색 노출 제한, 견적만 가능, 결제·공개 페이지 불가
  2. `docs_verified` — 사업자 + 기획업 등록 확인
  3. `artist_confirmed` — 아티스트 본인 승인 완료 (최고 등급 배지)

### L3. 거래 방어 (사기의 이득 차단)
- **에스크로 기본값**: 계약금 플랫폼 예치 → 이행 확인 후 지급. 잠적해도 수취 불가
- 신규 소속사 첫 거래 한도 (예: 건당 500만원) + 에스크로 강제
- 채팅 내 계좌번호·외부 연락처 패턴 감지 → "플랫폼 밖 송금은 보호받을 수 없습니다" 경고
  (탈중개 방지 겸용)

### L4. 운영 시그널
- 신고 버튼 (프로필/요청/채팅) → 관리자 검수 큐
- 이상 징후 자동 홀드: 신규 계정의 유명인 등록, 시세 대비 저가 미끼 견적, 단기 대량 등록
- 프로필 사진 유사도·역검색 도용 탐지 (P2)
- `booking_status_history` + 견적 이력 = 분쟁 증거 (이미 구현된 감사 로그 활용)

## 우선순위

| 단계 | 항목 | 비고 |
|---|---|---|
| **P0** (공개 페이지 발행 전 필수) | 신뢰 3단계 enum + 사업자/기획업 등록번호 필드 + 미검증 공개 페이지 발행 불가 + 신고 버튼 | Sprint 1에 포함 |
| P1 | 아티스트 본인 승인 알림톡 플로우, 첫 거래 한도, 국세청 API 자동화 | 인증 도입과 함께 |
| P2 | 채팅 직거래 감지, 사진 역검색, 이상 탐지 룰 | 데이터 쌓인 후 |

## 스키마 반영 예정
- `agencies`: `businessRegNo`, `entertainmentLicenseNo`, `trustLevel` enum(unverified/docs_verified/artist_confirmed), `verifiedAt`
- `artists`: `confirmedByArtistAt` (본인 승인 시각), `publishable` (공개 페이지 발행 가능 여부 — trustLevel 파생)
- `reports` 테이블: reporterId, targetType(profile/request/message), targetId, reason, status
