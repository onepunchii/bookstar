# 신기능 리서치 & 로드맵 제안 (2026-07-07)

4가지 아이디어 검토: ① 캘린더+날씨 ② 기사/여론 시각화 ③ Chartmetric·Songstats·버블·Artist Growth 벤치마킹 ④ 아티스트 개인 홈페이지.
결론: **4개 전부 실현 가능하고, 서로 연결하면 "인입 → 설득 → 실행" 플라이휠이 된다.**

## ① 캘린더 + 날씨 연동

### 데이터 소스 (리서치 결과)
- **기상청 공공데이터포털**: 무료, 운영계정 일 100,000콜. 단기예보(3~4일, 1시간 단위) + 중기예보(+11일).
- 11일 이후 행사일: **기후 평년값**("이 날짜 과거 30년 평균 최고 28℃, 강수확률 40%") 표시 → D-10 진입 시 실제 예보로 자동 전환 + 알림. 이벤트 플랫폼 표준 패턴.
- UI에서 "예보" vs "과거 평균 참고치" 라벨 구분 필수.

### 구현 포인트
- `schedules`/`daySchedules`/`holds` 테이블이 이미 있음 → 날씨는 `bookingRequests.location` 지오코딩(격자 좌표 변환) + 캘린더 셀/데이시트에 날씨 뱃지만 얹으면 됨.
- **야외 행사 우천 리스크 알림**: 확정 일정 D-7/D-3에 강수확률 임계치 초과 시 알림 → 매니저의 "펑크 공포"(RESEARCH.md 핵심 발견 1)를 직접 타격.
- **캘린더 외부 연동**: Google Calendar 양방향은 운영 복잡도 높음(웹훅 7일 만료, OAuth 민감 스코프 심사) → MVP는 **.ics 구독 피드**(반나절 구현, 무료)로 시작. 양방향은 유료 플랜 차별화로 후순위.

## ② 기사/여론 노출 빈도 시각화 ("화제성 대시보드")

### 스크래핑 대신 공식 API 3종 (전부 무료)
| 소스 | 내용 | 쿼터 |
|---|---|---|
| 네이버 뉴스 검색 API | 셀럽별 기사 건수 시계열 (본문 저장 X, 카운트만) | 일 25,000콜 — 셀럽 500명 일 1회 갱신 = 쿼터 2% |
| 네이버 데이터랩 | 검색어 트렌드 (상대값 0~100 → 앵커 키워드로 정규화 필요) | 일 ~1,000콜 |
| YouTube Data API v3 | 채널 구독자·조회수 (channels.list 1유닛, 50채널 배치) | 일 10,000유닛 — 500채널 = 쿼터 0.1% |

- **Instagram**: Business Discovery로 크리에이터 계정 팔로워 수 조회 가능(무료). 단 Meta 앱 리뷰 수 주 소요 → 2단계.
- **TikTok**: 공식 경로 없음(Research API는 학술 전용, Display API는 본인 계정만) → 수동 입력 or 보류. 직접 스크래핑은 B2B 신뢰도·ToS 리스크로 비추천.

### 스코어 설계 — Chartmetric Artist Rank 벤치마킹
"Fan Base(장기 팬덤 규모) vs Engagement(최근 화제성)" 이원화가 광고주 섭외 의사결정 프레임과 정확히 일치:
- **팬덤 지수**: 유튜브 구독자 + 인스타 팔로워 (절대 규모, 누적)
- **화제성 지수**: 최근 30일 기사 건수 증감 + 검색 트렌드 기울기 + 조회수 증가율
- 아티스트 상세 페이지에 스파크라인 그래프 → 소속사엔 "단가 협상 근거", 광고주엔 "섭외 확신" 제공.
- 일별 스냅샷 테이블(`artist_metrics`: artistId, date, newsCount, searchTrend, ytSubs, ytViews, igFollowers) 축적 → **시계열 데이터 자체가 시간이 지날수록 해자**.

## ③ 레퍼런스 4사 핵심 기능 이식 가능성

### Chartmetric ($117~150/월, API $350/월~)
- 킬러: Artist Rank 합성 점수, 플레이리스트 트래킹, 브랜드-아티스트 매칭.
- **한국 커버리지: Melon 차트 수준. Songstats는 한국 제로.** → 써클차트+국내 소셜 묶은 "국내 섭외용 스코어"는 글로벌 플레이어가 못 만드는 상품 = 우리 공백 시장.
- Spotify API는 2025~26년 신규 스타트업에 사실상 폐쇄(MAU 25만+ 요건) → 글로벌 스트리밍 지표는 재현하려 하지 말 것. 필요해지면 Songstats Enterprise API(볼륨 견적) 통합.

### Songstats (€11.99~$99/월)
- 교훈: "ID 하나로 전 플랫폼 통합 조회" UX + **실시간 활동 푸시 알림**(차트인 순간 알림 = 리텐션 견인) → "담당 아티스트 기사 급증 알림"으로 번역 가능.

### 디어유 버블 (월 4,500원/아티스트, 영업이익률 34%)
- 교훈 1 — **공급 라인업이 곧 마케팅**: 25개 기획사 개방형 입점이 성장 엔진. BOOKSTAR도 기획사 IP 확보가 수요 유입의 전제.
- 교훈 2 — **락인의 제품 내장**: 구독 유지일수록 답장 글자수 증가 → B2B 번역: 거래 실적 기반 고객사 등급(수수료 할인·우선 응답권).
- 교훈 3 — 수익 배분 40(플랫폼)/30(마켓)/30(기획사) 투명 구조가 공급자 참여 유인.

### Artist Growth ($9~14/seat/월)
- **Aggregated Day View**(하루치 일정·파일·연락처 자동 데이시트) → 우리 `daySchedules`가 이미 이 방향. 여기에 날씨를 얹으면 AG에도 없는 기능.
- **읽기 전용 시트 무제한 무료** → 광고주/현장 스태프에게 일정 read-only 공유 링크 = 바이럴 + 신뢰 장치 (RESEARCH.md P2-8과 일치).
- 충돌 감지(더블부킹 차단)는 AG도 명시 기능 아님 → 우리 홀드 시스템과 결합 시 차별화.
- 가격 레퍼런스: per-seat $9~14/월이 시장 수용가.

## ④ 아티스트 개인 홈페이지 (원페이지 EPK)

- 해외 부킹 SaaS(Gigwell 등)의 검증된 인입 채널 = **부킹 위젯 + EPK**. RESEARCH.md P0-1 "섭외 인입 통합"의 실행 수단.
- 구현: `artists` 테이블에 이미 프로필·갤러리·태그·SNS·recentWork 다 있음 → 소속사가 사진만 올리면 `bookstar.kr/@슬러그` 공개 페이지 자동 생성.
  - 구성: 히어로 사진 + 카테고리/태그 + ②의 화제성 그래프 + ①의 가능 일정 캘린더(가능/불가만) + **"섭외 문의" CTA → 우리 인박스로 직행**
  - OG 이미지 자동 생성(@vercel/og) → 카톡 공유 시 예쁜 카드
  - 템플릿 2~3종 + 소속사 로고
- 전략적 의미: 소속사가 이 링크를 **자기 명함/SNS 프로필에 걸고 다니는 순간 BOOKSTAR가 공식 섭외 창구**가 됨 = CAC 0의 인입 채널 (버블의 "팬덤 따라 유입" 구조의 B2B 버전). SEO로 "○○○ 섭외" 검색 유입까지.

## 통합 로드맵 (플라이휠: 인입 → 설득 → 실행)

```
④ 개인 홈페이지(인입) → ② 화제성 그래프(섭외 설득) → ① 캘린더+날씨(실행/운영) → 정산(기존)
```

### Phase 1 (2~3주) — 전부 무료 API
1. ④ 아티스트 공개 페이지 (`/@slug`) + 섭외 CTA + OG 이미지
2. ② 네이버 뉴스 카운트 + YouTube 구독자 수집 배치(Vercel Cron) + `artist_metrics` 테이블 + 스파크라인
3. ① .ics 구독 피드 export

### Phase 2 (3~4주)
4. ① 기상청 날씨 → 캘린더/데이시트 뱃지 + 우천 리스크 알림
5. ② 데이터랩 검색 트렌드 + 화제성/팬덤 지수 산식 + 대시보드 그래프
6. ③ 일정 read-only 공유 링크 (AG 패턴)

### Phase 3 (이후)
7. ② Instagram Business Discovery (Meta 앱 리뷰 통과 후)
8. ② "화제성 급증 알림" 푸시 (Songstats 패턴)
9. ① Google Calendar 양방향 동기화 (유료 플랜 차별화)
10. ③ 고객사 등급/락인 설계 (버블 패턴)

### 하지 말 것
- 플랫폼 직접 스크래핑 (인스타/틱톡 웹) — ToS·차단 리스크, B2B 신뢰도 훼손
- Spotify/글로벌 스트리밍 지표 자체 구축 — API 폐쇄로 불가능, 필요시 Songstats API 구매
- 초기부터 Google Calendar 양방향 — .ics로 충분
